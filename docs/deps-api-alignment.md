We need to harmonize **gpui-component**, **leptos-shadcn-ui** and **shadcn-dioxus** – three ShadCN‑style UI libraries for different Rust frameworks – so that we can later generate any component via a canonical macro, without per‑library polyfills.  
This analysis compares every component, identifies discrepancies (missing pieces, naming clashes, state‑management differences) and proposes the minimal changes each library should accept to converge toward the original **shadcn/ui** (React) API.

---

## 1. Global Mapping of Frameworks

| Feature            | gpui-component                                                                    | leptos-shadcn-ui (umbrella)                                                        | shadcn-dioxus (ui crate)                                                          |
| ------------------ | --------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------- | --------------------------------------------------------------------------------- |
| State management   | `Entity<T>` (state handles created via `cx.new`), subscriptions                  | `RwSignal<T>`, `Signal<T>`, `MaybeProp<T>` (Signal<Option<T>>), `Callback<T>`      | `Signal<T>`, `Option<Callback<T>>`, context via `use_context`                     |
| Prop definition    | Builder methods on the component struct (`.primary()`, `.icon()`, `.on_click()`) | `#[component]` with `#[prop(...)]` attributes, often `MaybeProp<String>` for class | `#[derive(Props)]` struct with `#[props(default)]`, owned `String`/`Vec<T>`       |
| Children structure | Fluent `.child(...)` / `.children(...)`, items built by closures                 | Nested components: `<Accordion><AccordionItem><AccordionTrigger>…`                  | Same nested components as Leptos, using `children: Element`                       |
| Overlays           | `Root` view renders layers for dialogs, sheets, notifications                    | Portal‑based; requires a `<PortalProvider>` wrapper (implied)                       | Portal‑based; requires `<PortalProvider>`                                          |
| Theme / assets     | `ActiveTheme` trait, custom asset source                                         | Tailwind CSS classes baked in                                                      | Tailwind CSS classes baked in                                                     |
| Variant naming     | `primary`, `danger`, `ghost`, `link`, `text`, `info`/`success`/`warning`/`error` | `Default`, `Destructive`, `Secondary`, `Ghost`, `Link`, `Outline`, `Success`, etc. | `Default`, `Destructive`, `Secondary`, `Ghost`, `Link`, `Outline` (no `Success`)  |

| Callback arguments | `(&mut Window, &mut Context<T>)` plus relevant value                             | Value only via `Callback<T>`                                                        | Value only via `Option<Callback<T>>`                                               |

## 3. Overlay / Portal Layer Differences

- **gpui‑component** uses a special `Root` view that renders a stack of dialogs, sheets, and notifications. The API is imperative (`window.open_dialog`) or declarative (`Dialog::new(cx)…`).  
- **leptos‑shadcn‑ui** and **shadcn‑dioxus** rely on a `PortalProvider` that wraps the entire app and portals overlay content into it. Dialogs/Drawers/Popovers are controlled via `open` signals.

For macros, the canonical spec must abstract the overlay system. It is easiest if all libraries support both controlled (`open` signal) and uncontrolled (`default_open` + callbacks) modes. gpui already has declarative dialogs; other two already have signals.

## 4. Global Patterns Requiring Alignment

1. **Variant and Size Naming**  
   Unify on the original ShadCN names:
   - Button variants: `Default`, `Destructive`, `Outline`, `Secondary`, `Ghost`, `Link`
   - Alert variants: `Default`, `Destructive`, `Success`, `Warning`
   - Sizes: `Default`, `Sm`, `Lg`, `Icon` (plus `IconXs`, `IconXl` where needed)
   - gpui’s `primary` → `default`, `danger` → `destructive`, `text` → `link`

2. **Callback Signatures**  
   Use `on_change: Callback<T>` (or `Option<Callback<T>>`) standardized across libraries. In gpui, the macro can adapt the closure to fit `(&mut Window, &mut Context<T>)`.

3. **State vs Signal**  
   In the canonical spec, a component that manages state internally can be used with a controlled `value`/`checked` signal or an uncontrolled `default_value`/`default_checked` property. The macro can decide which pattern to generate per framework.

4. **Child Composition**  
   The canonical spec should define components as a tree (like the original JSX). Frameworks that use a builder pattern (gpui) will need the macro to translate nested children into fluent method calls. This is complex but feasible if we limit the canonical spec to a subset that both structures can express. Ideally, all libraries adopt the same pattern – **nested components** (like the original ShadCN). gpui can move toward that pattern (it already supports `.child(SomeElement)`), so we only need to provide the corresponding sub‑components.

5. **Missing Components & Sub‑components**  
   The matrix shows clear gaps. Leptos‑shadcn‑ui lacks many components; shadcn‑dioxus is missing some variants and sub‑components; gpui‑component misses several core ShadCN components (Card, Command, Menubar, etc.). The first step is to fill these gaps so the canonical set is available everywhere.

## 5. Proposed Harmonization Roadmap (Library Contributions)

### For **leptos-shadcn-ui** (highest number of gaps)
- Add missing components: `ColorPicker`, `DataTable`, full `DropdownMenu`, `Empty`, `HoverCard`, `Menubar`, `NavigationMenu`, `Popover`, `ScrollArea`, `Sidebar`, `Spinner`, `Stepper`, `Kbd`.
- Implement proper handling for Drawer/Sheet (already present).
- Ensure all components accept `Signal<bool>` for `open`/`checked` etc., matching the controlled pattern.
- Provide `RwSignal`-based state for components that need it (Calendar, Select, etc.).

### For **shadcn-dioxus** (minor gaps)
- Add `Success`/`Warning` variants to `Alert`.
- Add `BreadcrumbEllipsis`.
- Add `AvatarGroup`.
- Add `CardVariant` (Default, Destructive, etc.).
- Add `loading` prop to `Button`.
- Improve `Progress` with variants and label (already has value/max).
- Consider adding a `Spinner` if not present (already there).
- Add `KbdGroup` (Kbd already there).
- Ensure `DataTable` is fully featured (it seems so).

### For **gpui-component** (larger structural differences)
- **Crucial**: Transition from builder‑only with internal state to nested components where possible. For example, `Accordion` should be composable as `AccordionItem` / `AccordionTrigger` / `AccordionContent`. This can be done additively while keeping old API.
- Introduce `AccordionItem`, `AccordionTrigger`, `AccordionContent` sub‑components.
- Provide `AvatarImage` and `AvatarFallback` children instead of just `.src()`.
- Add string‑based value for `RadioGroup` and `Tabs`.
- Implement missing core components: `Card`, `Carousel`, `Combobox`, `Command`, `Empty`, `Menubar`, `NavigationMenu`, `ScrollArea`, `Separator`.
- Standardise variant names (`primary` → `default`, etc.).
- Add a `loading` prop to `Button`.
- Add indeterminate state to `Checkbox`.
- Expose a `controlled` API for components that currently rely on `Entity` state (e.g., `Calendar` with an `on_select` callback and optional `selected` signal). This can live alongside the stateful API.
- Improve overlay handling to optionally support signal‑driven `open` for dialogs/sheets (already partially done with declarative API).

These changes would reduce the polyfilling burden dramatically and make the canonical macro a simple mapping layer rather than a full translation engine.

## 6. Canonical API Specification (Draft)

For macro input, we can design a JSON/YAML/Macro‑DSL that describes any component. Example for `Button`:

```rust
canonical_component! {
    name: "Button",
    props: {
        variant: ButtonVariant = Default,
        size: ButtonSize = Default,
        disabled: bool = false,
        loading: bool = false,
        on_click: Option<Callback<()>>,
    },
    slot: "default" (children),
}
```

- **Enums** are predefined (e.g., `ButtonVariant::Default | Destructive | …`).
- **Callbacks** follow `Option<Callback<T>>`.
- **Slots** allow named children (e.g., `trigger`, `content` for `Tooltip`).

During expansion, the macro calls framework‑specific impl blocks that map these to the actual library API. Alignment of the libraries would let us keep those impl blocks trivial.

## 7. Next Steps

1. **Open issues / PRs** in all three repositories referencing this analysis to start filling the gaps and aligning the APIs.  
2. **Design the canonical spec** formally and share with the authors to get buy‑in.  
3. **Implement the macro crate** that uses the spec and generates code for gpui, leptos, and dioxus. The macro will seamlessly handle any remaining polyfills (e.g., gpui’s entity state ↔ signal).  
4. **Provide migration guides** where existing APIs change (especially for gpui’s variant naming).

This harmonization effort will make `leptos-shadcn-ui`, `shadcn-dioxus` and `gpui-component` truly interchangeable at the source level, saving enormous effort for cross‑framework UI development.
